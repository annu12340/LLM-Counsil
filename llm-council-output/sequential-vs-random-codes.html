⚖ LLM Council Debate

Sequential vs Random Codes for tiny-shortener

A structured, adversarial debate among five expert personas, synthesized into a decisive recommendation.

Single-model baseline — no council

Yes, you should generally prefer random or hashed short codes over sequential IDs. Sequential IDs are enumerable — anyone can walk /1, /2, /3 and discover every link — and they leak how many URLs you've created. Random codes (or a hash of the URL) avoid this and are considered a best practice for URL shorteners. Just be mindful of collisions and pick a sufficient code length.

Why this is risky: It's ungrounded in this repo (in-memory, single-run, no network surface to enumerate), it hedges instead of committing, and it names no concrete cost — collision handling, the persistence prerequisite, or the fact that hashing the URL would break the per-code _clicks feature.

Topic

Should tiny-shortener switch from sequential ids to random/hashed codes?

Context

The topic points at High critical project mock — the tiny-shortener fixture. Shortener holds two in-memory dicts (_urls, _clicks) plus a _next_id counter starting at 1. shorten() returns encode(_next_id) — the base-62 of a monotonic counter (1→"1", 62→"10") — then increments. The CLI builds a fresh Shortener per invocation, so the README confirms codes do not persist between commands.

What does NOT exist: no database or persistence, no HTTP/network layer, no authentication, no collision detection, no rate limiting. test_unique_codes passes trivially because a monotonic counter can't repeat.

So two questions are tangled, and the council addresses both: the literal question (sequential vs random/hashed generation) and the premise — in a process that resets every run with no network, does the choice change anything today?

Inferred decision criteria
1. Enumeration / privacy — can an outsider discover or count links from the codes?
2. Collision & correctness cost — what new failure modes does each scheme introduce?
3. Fit to current architecture — does the change solve a problem this code has now?
4. Migration cost / reversibility — how expensive is choosing wrong later?

Assumption: "switch" implies intent to grow this toward a real, networked service; otherwise the question is inert.

1 Round 1 — Opening Positions

🧱First-Principles Thinker: A short code is fundamentally a public handle pointing to a stored URL. The defect in sequential ids isn't the math — it's that they fuse the public handle to an internal monotonic counter, leaking two facts for free: total link count and the address of every neighbor (n±1). The correct principle for any real shortener is to decouple the public code from the internal id. But this repo has no store and no network — no corpus to enumerate, no counter that survives one run — so the principle is right while the literal switch buys nothing here yet.
Verdict: Decoupling code from internal id is correct — but moot until there's persistence; neither now.
🔬Research Scientist: These are not one choice but three. (a) Sequential: zero collisions by construction. (b) Random base-62 of length L: collision follows the birthday bound — for L=7, space ≈ 62⁷ ≈ 3.5×10¹², first expected collision near √space ≈ 1.9M inserts, so you must add a uniqueness-check-and-retry against the store. (c) Hashing the URL (truncated SHA): free dedup (same URL → same code) but preimage-enumerable for known URLs and still needs truncation-collision handling. Different properties — "random/hashed" should not be treated as one option.
Verdict: Random is sound but needs collision-retry; hashing-the-URL is a different feature — don't conflate them.
⚙️Systems Engineer: Mechanically, sequential is O(1), unique by construction, no extra read. Random requires a "does this code exist?" check on every insert — free against the current in-memory dict, but in a real datastore it's an extra round-trip plus a unique-index-conflict retry loop. The honest observation: the architecture has no datastore and no server, so changing the code generator is rearranging deck chairs. The move that matters is generating codes at the persistence boundary you don't have yet.
Verdict: Don't touch code generation until a datastore exists; neither now.
🚩Skeptic / Red Team: The premise is the real finding. Sequential ids in a deployed shortener are a genuine, cheap exfiltration vector: a curl loop over /1..N scrapes every URL anyone shortened and reveals total volume — real business and privacy harm. But this code has zero attack surface: in-memory, single process, no network, resets on exit. The danger isn't the id scheme; it's shipping this toy as-is and inheriting sequential ids as a data-leak. And random codes are necessary-not-sufficient — short random codes are still brute-forceable without rate limiting and length.
Verdict: Random the moment it's networked — but the live risk is treating a toy as production, and random alone isn't a security control.
👤Product / User Advocate: Users want clean, shareable links that don't broadcast "you're customer #3." https://sh.rt/1 looks broken and lets anyone count your usage; https://sh.rt/aB3xK reads like a real shortener and protects privacy. So random codes win on product polish and privacy — but keep them short (6–7 chars). And a sharp warning grounded in the code: do not hash the URL — two users shortening the same URL collide onto one code and share _clicks, silently destroying per-link analytics, a feature this repo already has.
Verdict: Random short codes for polish + privacy; explicitly NOT hashed-from-URL — it breaks per-code click stats.

The codes don't survive one CLI call — a user can't even share sh.rt/aB3xK because it won't resolve in the next process. Cosmetics are downstream of persistence; ranking polish first inverts the dependency order.
Verdict: Neither now — code aesthetics are downstream of the persistence this repo lacks.
🔬Research Scientist → Skeptic: You called sequential a "data-leak" but then said this repo has zero attack surface — agreed, yet you under-specify the fix. "Random the moment it's networked" skips the collision-retry cost and the length math. If someone takes that literally and generates 5-char random codes with no uniqueness check, they ship duplicate codes that overwrite live URLs. The fix needs a number and a retry loop, not just "make it random."
Verdict: Random requires explicit length + collision-retry, not a bare "make it random."
⚙️Systems Engineer → Research Scientist: Your taxonomy is correct but you're treating collision-retry as a real cost — against the current dict it's a single in check, effectively free, and even in Postgres it's a unique-index conflict that's rare at L≥7. Don't inflate the cost of the right design; the actual blocker isn't collisions, it's that there's nothing to persist into.
Verdict: Collision cost is negligible at L≥7; the blocker is missing persistence, not retries.
🚩Skeptic / Red Team → Systems Engineer: "Rearranging deck chairs" undersells one thing: which code scheme you bake in now is the one you're stuck migrating later. If you persist sequential codes first and go live, switching to random means either breaking every old short URL or running both schemes forever. Deferring the datastore is right; defaulting the code scheme to sequential "for now" quietly sets a trap.
Verdict: Defer the datastore, but pick random as the scheme so you never migrate live URLs.
👤Product / User Advocate → First-Principles Thinker: Fair that persistence comes first — but "neither, it's moot" leaves the developer with no decision for the moment they do add a store, which is clearly their trajectory ("switch"). Give them the rule to apply at that boundary, not just a "not yet."
Verdict: Random short codes — and give the trigger condition, not a bare "not yet."

⚖ Final Verdict

Best argument
The First-Principles Thinker's, reinforced by the Skeptic — the issue isn't base-62 math, it's that sequential ids fuse the public handle to an internal monotonic counter. Deciding the scheme now (even though today's in-memory behavior is unaffected) avoids the migration trap of going live on sequential codes.
Biggest risk
Migration trap. Persisting sequential codes and shipping, then being forced to either break every existing short URL or run two code schemes forever. Secondary: shipping the toy as-is and inheriting enumeration + a brute-forceable namespace.
Key tradeoff
Random decouples the public code from the internal id (no enumeration, no count leak) at the cost of a uniqueness-check-and-retry on insert and a chosen length. Sequential is collision-free and O(1) but leaks count and is fully enumerable. Hashing the URL is a different feature (dedup) that breaks per-code analytics and is preimage-enumerable — rejected.
Consensus
All five agree the literal switch changes nothing in the current in-memory, no-network toy, and that persistence is the real prerequisite. All five also agree that when persistence/networking is added, the public code should be random, not sequential — decided now, applied then. Unanimous rejection of hashing the URL as the code.
Recommendation

Don't change runtime behavior today, but commit to random codes as the scheme. When you add persistence (the actual next step), generate the public code as random base-62, length ≥7, with a uniqueness check-and-retry against the store, kept separate from the internal id. Do not derive the code from a hash of the URL — it breaks the per-code _clicks feature and is preimage-enumerable. Before exposing resolve over a network, add rate limiting — random codes prevent enumeration but not brute force.

📋 ADR Snippet

Status: proposed

Decision: Adopt random base-62 codes (length ≥7, generate-and-check for uniqueness) as the public short-code scheme, decoupled from the internal id and applied when persistence is introduced; reject sequential ids and reject hashing the URL into the code.

Consequences: Buys non-enumerable, count-hiding, professional-looking links and avoids a future live-URL migration. Costs a uniqueness-check-and-retry on insert and a committed code length. Makes per-code analytics safe (no URL-hash collisions), but does not by itself stop brute force — rate limiting becomes a prerequisite before public exposure. No behavior change in the current in-memory CLI.

Bottom line
For tiny-shortener as it exists today, switching the id scheme changes nothing — it's in-memory, single-run, and has no network to enumerate. But the answer to the design question is a clear yes, go random: decide it now so you never persist sequential codes and inherit a migration trap. When you add a datastore, generate random base-62 codes (≥7 chars) with a uniqueness retry, keep them separate from the internal id, and add rate limiting before exposing resolve. Do not hash the URL — that silently merges per-link click counts.
